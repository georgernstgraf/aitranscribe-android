# AITranscribe Android Architecture Documentation

## Overview

This document describes the complete architecture of the AITranscribe Android application. It serves as a reference for future agents, maintainers, and contributors.

**Design Principles:**
- **Test-First Development:** Tests written before implementation
- **Clean Architecture:** Separation of concerns (Data, Domain, UI)
- **Dependency Injection:** Using Hilt for loose coupling
- **Reactive Programming:** Flow-based data streams
- **SOLID Principles:** Single Responsibility, Open/Closed, Liskov Substitution

## 🏗️ Architecture Layers

### 1. Data Layer

**Purpose:** Manages data persistence, API communication, and external dependencies.

**Components:**

#### Local Database (Room)

**Entities:**
- `TranscriptionEntity` - Main transcription data
- `QueuedTranscriptionEntity` - Offline queue items
- `AppPreferenceEntity` - App preferences

**DAOs:**
- `TranscriptionDao` - Basic CRUD operations
- `TranscriptionDaoEnhanced` - Optimized queries with bulk operations
- `QueuedTranscriptionDao` - Offline queue management
- `AppPreferencesDao` - Preferences storage

**Database:**
- `TranscriptionDatabase` - Main database with migrations
- `TranscriptionDatabaseEnhanced` - Enhanced version

**Modules:**
- `DatabaseModule` - Provides DAO instances
- `SecurePreferences` - Encrypted API key storage

#### Remote (API Clients)

**Services:**
- `GroqApiService` - GROQ STT API client
- `OpenRouterApiService` - OpenRouter LLM API client

**DTOs:**
- `GroqTranscriptionResponse` - GROQ response model
- `OpenRouterRequest/Response` - OpenRouter models

#### Repository

**Pattern:** Repository Pattern (interface + implementation)

**Repositories:**
- `TranscriptionRepository` - Main transcription data
- `AppPreferencesRepository` - App preferences

**Module:**
- `RepositoryModule` - Provides repository instances

**Testing:**
- `FakeTranscriptionRepository` - In-memory test double

## 🔄 Data Flow

### Recording Flow

```
User Action (Push & Hold)
    ↓
RecordingService.start()
    ↓
MediaRecorder.start()
    ↓
Foreground Notification
    ↓
User Release or Cancel
    ↓
Audio file saved to temp
    ↓
TranscriptionWorker.enqueue()
    ↓
TranscriptionWorker.execute()
    ↓
GROQ API Call
    ↓
OpenRouter API Call (if enabled)
    ↓
Database.insert()
    ↓
Notification.complete()
```

### Search Flow

```
User Input (Date/Text/Filter)
    ↓
SearchViewModel.onFilterChanged()
    ↓
SearchTranscriptionsUseCase.invoke()
    ↓
TranscriptionRepository.searchTranscriptions()
    ↓
TranscriptionDao.searchTranscriptions()
    ↓
StateFlow.emit(SearchResults)
    ↓
SearchScreen.updateUI()
```

### Offline Queue Flow

```
NetworkMonitor.isDisconnected()
    ↓
TranscriptionWorker.enqueue()
    ↓
NetworkMonitor.isReconnected()
    ↓
WorkManager.execute(TranscriptionWorker)
    ↓
QueueOfflineTranscriptionUseCase.invoke()
    ↓
TranscriptionRepository.queueForOffline()
    ↓
Next network connection
    ↓
Process queue (FIFO)
```

## 🗄️ Database Schema

### Transcriptions Table

```sql
CREATE TABLE transcriptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    original_text TEXT NOT NULL,
    processed_text TEXT,
    audio_file_path TEXT,
    created_at TEXT NOT NULL,
    post_processing_type TEXT,
    status TEXT NOT NULL,
    error_message TEXT,
    played_count INTEGER NOT NULL DEFAULT 0,
    retry_count INTEGER NOT NULL DEFAULT 0
);

-- Indexes for performance
CREATE INDEX idx_transcriptions_played ON transcriptions(played_count);
CREATE INDEX idx_transcriptions_created ON transcriptions(created_at DESC);
CREATE INDEX idx_transcriptions_status ON transcriptions(status);
```

### Queued Transcriptions Table

```sql
CREATE TABLE queued_transcriptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    audio_file_path TEXT NOT NULL,
    stt_model TEXT NOT NULL,
    llm_model TEXT,
    post_processing_type TEXT,
    created_at TEXT NOT NULL,
    priority INTEGER NOT NULL DEFAULT 0
);

-- Ordered by creation time and priority
CREATE INDEX idx_queued_created ON queued_transcriptions(created_at ASC, priority DESC);
```

### App Preferences Table

```sql
CREATE TABLE app_preferences (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

## 🎯 Key Features Implementation

### 1. Push-to-Talk Recording

**Component:** `AudioRecordingButton`  
**Service:** `RecordingService`  
**Gesture:** `Modifier.pointerInput` with `detectTapGestures`  
**Audio:** `MediaRecorder` with AAC encoding  
**Foreground:** Foreground service with notification

### 2. GROQ STT Integration

**Use Case:** `TranscribeAudioUseCase`  
**Service:** `GroqApiService`  
**Audio Processing:** `AudioChunker` (25MB limit)  
**Error Handling:** Retry logic with exponential backoff

### 3. OpenRouter LLM Integration

**Use Case:** `PostProcessTextUseCase`  
**Service:** `OpenRouterApiService`  
**Prompt Types:** Grammar correction, English translation  
**Prompt Building:** `buildSystemPrompt()` method

### 4. View Status Tracking

**Database:** `played_count` field  
**Indicator:** Blue dot for unviewed (played_count = 0)  
**Use Cases:** `MarkAsViewedUseCase`, `ResetViewStatusUseCase`  
**Auto-Mark:** Incremented when opening detail view

### 5. Search & Filtering

**Use Case:** `SearchTranscriptionsUseCase`  
**Filters:** Date range, text search, view status  
**Default Filter:** `ViewFilter.UNVIEWED_ONLY`  
**DAO Query:** Optimized SQL with parameterized queries

### 6. Delete Operations

**Use Case:** `DeleteTranscriptionUseCase`  
**Modes:** Single, Old All, Old Unviewed  
**Confirmation:** Show count before deletion  
**UI Component:** `DeleteOldDialog`

### 7. Offline Queue

**Use Case:** `QueueOfflineTranscriptionUseCase`  
**Worker:** `TranscriptionWorker`  
**Monitoring:** `NetworkMonitor` with Flow  
**Retry:** Automatic on network reconnect

### 8. Background Processing

**Worker:** `TranscriptionWorker` (WorkManager)  
**Notifications:** `EnhancedNotificationManager`  
**Chains:** Recording → STT → LLM → Storage

## 📊 Performance Considerations

### Database Optimization

- Use indexed queries
- Batch operations for bulk inserts/updates
- Proper column types
- Query optimization with `@Query` annotations

### UI Performance

- Use `remember` for expensive computations
- Lazy loading for lists
- Avoid recomposition with proper state management
- Use `LaunchedEffect` for one-time operations

### Memory Management

- Release audio files after transcription
- Use `LazyColumn` for large lists
- Proper coroutine scoping
- Clear references in `onDispose`

### Network Efficiency

- OkHttp connection pooling
- Request/response caching where appropriate
- Offline queue to avoid failed requests
- Exponential backoff for retries

## 🐛 Error Handling

### Exception Hierarchy

```kotlin
TranscriptionException
    └── TranscriptionException (STT failures)
PostProcessingException
DeleteException
QueueException
ExportException
ImportException
ShareException
BackupRestoreException
```

### Error Handling Strategy

1. **Log error with AppLogger**
2. **Record error in database** (if applicable)
3. **Show user-friendly message**
4. **Offer retry option** (for transient errors)
5. **Notify user** (for background errors)

### Error Types

**Transient Errors:** Retry with exponential backoff
- Network timeout
- Temporary API errors
- Service unavailable

**Permanent Errors:** Show error message
- Invalid API keys
- Invalid audio format
- Storage errors
